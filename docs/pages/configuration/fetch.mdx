# Fetch Providers

The `fetch_webpage` tool fetches web page content for use in chat. Vortex uses a **provider chain** — it tries each provider in order and returns the first successful result.

## Provider chain

| Priority | Provider | Key required | When active |
|---|---|---|---|
| 1 | **Firecrawl** | `FIRECRAWL_API_KEY` | Only when key is set |
| 2 | **Crawl4AI** | None | When `crawl4ai` Python package is installed |
| 3 | **Jina Reader** | Optional (`JINA_API_KEY`) | Always |
| 4 | **requests** | None | Always (last resort) |

## Providers explained

**Firecrawl** — Hosted service. Best at bypassing Cloudflare and bot protection. Returns clean markdown. Requires an API key from firecrawl.dev.

```yaml
fetch:
  firecrawl_api_key: "fc-..."
```

**Crawl4AI** — Local Playwright-based scraper with stealth mode. Handles JavaScript-heavy pages without an API key. Active only if the `crawl4ai` package is installed in the Python environment:

```bash
pip install crawl4ai
playwright install chromium
```

**Jina Reader** — Proxies requests through `r.jina.ai` to return clean markdown. Always active; no key required. Set `JINA_API_KEY` to increase rate limits:

```yaml
fetch:
  jina_api_key: "jina_..."
```

**requests** — Plain HTTP GET. Always present as a last resort. Fails on JavaScript-heavy pages and Cloudflare-protected sites.

## Recommended setup

For production with reliable scraping:

```yaml
fetch:
  firecrawl_api_key: "fc-..."    # best bypass, handles Cloudflare
  jina_api_key: "jina_..."       # higher rate limits as fallback
```

No configuration is required beyond setting keys — Vortex builds the chain automatically.
