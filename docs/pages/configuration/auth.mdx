# Auth & Deployment Modes

Vortex has three deployment modes controlled by `DEPLOYMENT_MODE` (or `server.deployment_mode` in config.yaml).

import { Callout } from 'nextra/components'

## Deployment modes

### `dev` ‚Äî local development only

```yaml
server:
  deployment_mode: dev

auth:
  dev_bearer_token: devtoken
  dev_seed_user_email: dev@localhost
```

- All API routes accept a fixed bearer token (`DEV_BEARER_TOKEN`)
- A seed user is auto-created on startup
- No `SECRET_KEY` required
- CORS is fully open (`allow_origins: ["*"]`)
- **Never use in production**

### `selfhosted` ‚Äî single organization, invite-only

```yaml
server:
  deployment_mode: selfhosted

auth:
  secret_key: "your-secret-key-here"

smtp:
  host: smtp.example.com
  # ...
```

- Requires `SECRET_KEY` for JWT signing
- On first boot with an empty database: **all API routes return `503 Setup Required`** except `/health`, `/setup`, and `/auth/login`
- Admin completes setup at `/setup` to create the org and first user
- After setup: invite-only registration
- SMTP required for email verification and password reset

#### First-boot flow

```
Deploy with DEPLOYMENT_MODE=selfhosted
        ‚Üì
All API routes ‚Üí 503 (except /health, /setup, /auth/login)
        ‚Üì
Navigate to https://your-app/setup
        ‚Üì
Create org name + admin email + admin password
        ‚Üì
Instance live ‚Äî log in with admin credentials
```

### `saas` ‚Äî open signup, multi-tenant

```yaml
server:
  deployment_mode: saas

auth:
  secret_key: "your-secret-key-here"

smtp:
  host: smtp.example.com
  # ...
```

- Requires `SECRET_KEY`
- Open registration via `/auth/register` and `/auth/login`
- SMTP required for email verification
- No setup wizard ‚Äî users self-register

## Microsoft Entra (legacy)

`AUTH_MODE=entra` enables JWT validation via Microsoft Entra (Azure AD). This is the legacy auth mode for enterprise deployments.

```yaml
auth:
  mode: entra
  entra_tenant_id: "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
  entra_api_audience: "api://your-api-client-id"
  portal_api_key_pepper: "long-random-secret"
```

<Callout type="warning">
`auth.portal_api_key_pepper` is **required** when `auth.mode=entra`. It is used as an HMAC key to hash portal API keys (`aip_‚Ä¶`). Set a long random secret.
</Callout>

Frontend env vars for Entra:
```
VITE_AUTH_MODE=entra
VITE_ENTRA_SPA_CLIENT_ID=<SPA app registration client ID>
VITE_ENTRA_TENANT_ID=<tenant UUID>
VITE_ENTRA_API_SCOPE=<full scope string, e.g. api://<id>/access_as_user>
```

Set `auth.entra_debug_jwt=true` to include JWT error details in 401 responses during local debugging. Never enable in production.
