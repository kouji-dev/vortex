# Upgrading

How to upgrade Vortex to a new version.

import { Callout } from 'nextra/components'

## Docker Compose

```bash
# Pull latest code
git pull

# Rebuild images
docker compose --profile full build

# Restart — migrations run automatically on container start
docker compose --profile full up -d api web
```

Database migrations run automatically via the Dockerfile CMD (`alembic upgrade head`). No manual migration step needed.

### After major upgrades

Check the release notes for the new version. If the changelog mentions new model catalog entries:

```bash
docker compose --profile full run --rm api seed-catalog-models
```

## Manual install

```bash
git pull
cd backend
source .venv/bin/activate
pip install -e .
alembic upgrade head
seed-catalog-models    # if new models added
```

Restart the API process (e.g. `systemctl restart vortex-api`).

For the frontend:
```bash
cd frontend
pnpm install
pnpm build
```

Restart the frontend process.

<Callout type="info">
`config.yaml` changes never require restarting the process manually — Vortex reads settings fresh on each request. However, a restart is needed when changing `SECRET_KEY`, `DATABASE_URL`, or `DEPLOYMENT_MODE`.
</Callout>

## Rollback

To roll back to a previous version:

```bash
git checkout <previous-tag>
docker compose --profile full build
docker compose --profile full up -d api web
```

<Callout type="warning">
If the upgrade ran forward Alembic migrations, rolling back the code does **not** automatically downgrade the schema. Run `alembic downgrade <revision>` manually if needed, or restore from a backup.
</Callout>
