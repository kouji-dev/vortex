# Docker Compose

Recommended installation method. Runs Vortex in Docker containers with all dependencies managed.

import { Callout, Steps } from 'nextra/components'

## What this sets up

The `docker-compose.yml` defines four services:

| Service | Profile | Description |
|---|---|---|
| `db` | (default) | PostgreSQL 17 with pgvector |
| `redis` | (default) | Redis 7 |
| `api` | `full` | FastAPI backend |
| `web` | `full` | React frontend |

Run `docker compose up -d` (no profile) to start only the database and Redis — useful when running the API and frontend on the host for development. Add `--profile full` to also start the API and frontend in Docker.

<Steps>

### Clone

```bash
git clone https://github.com/your-org/vortex.git
cd vortex
```

### Create `config.yaml`

Place at the **repo root** (next to `docker-compose.yml`). The API container mounts this file via `AI_PORTAL_CONFIG`.

Full reference: [Configuration Reference →](/configuration/reference)

Minimal production example:

```yaml
server:
  deployment_mode: selfhosted
  cors_origins: https://your-frontend-domain.com

database:
  url: postgresql+psycopg://postgres:postgres@db:5432/ai_portal

auth:
  secret_key: "your-secret-key-here"

smtp:
  host: smtp.resend.com
  port: 587
  user: resend
  password: "re_..."
  email_from: noreply@yourdomain.com

llm:
  anthropic_api_key: "sk-ant-..."

embedding:
  voyage_api_key: "pa-..."
```

<Callout type="info">
Note `database.url` uses `@db:5432` (the Docker service name) not `@127.0.0.1:5434` (the host-mapped port). Use the Docker service hostname inside containers.
</Callout>

### Start Postgres and Redis

```bash
docker compose up -d
docker compose ps
```

Expected output: `db` and `redis` services with status `running`.

### Build and start the API and frontend

```bash
docker compose --profile full build
docker compose --profile full up -d api web
```

Migrations run automatically via the Dockerfile CMD:
```
alembic upgrade head && uvicorn ai_portal.main:app ...
```

### Seed the model catalog

Run once after first deploy, and again when upgrading to a version that adds new models:

```bash
docker compose --profile full run --rm api seed-catalog-models
```

<Callout type="warning">
If `seed-catalog-models` fails with a validation error, run with `--skip-model-validation` to bypass the API reachability check (useful when running offline or with a custom model registry):
```bash
docker compose --profile full run --rm api seed-catalog-models --skip-model-validation
```
</Callout>

### Run the setup wizard (selfhosted mode only)

Navigate to `http://localhost:3000/setup` and create your organization and admin account.

If `DEPLOYMENT_MODE=saas`, users can register directly — no setup wizard.

</Steps>

## Verifying the install

```bash
# API health
curl http://localhost:8000/health

# Confirm migrations ran
docker compose logs api | grep "alembic"

# Check model catalog was seeded
curl http://localhost:8000/api/model-catalog | python -m json.tool | head -20
```

## Ports

| Service | Host port | Container port |
|---|---|---|
| Postgres | 5434 | 5432 |
| Redis | 6380 | 6379 |
| API | 8000 | 8000 |
| Frontend | 3000 | 3000 |

These host ports are chosen to not conflict with other common local services. The API and frontend ports can be changed via `config.yaml` (`server.port`) and the `web` service's `PORT` env var.

## Resetting the database

```bash
docker compose down -v          # removes volumes (destructive!)
docker compose up -d
docker compose --profile full run --rm api alembic upgrade head
docker compose --profile full run --rm api seed-catalog-models
```
