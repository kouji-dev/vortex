# Overview

Vortex is a self-hostable AI portal — a web application that gives your team a shared interface for chat, knowledge bases, and model governance.

## What it includes

- **Chat** — streaming conversations with any LLM (Anthropic Claude, OpenAI GPT, Google Gemini, or any OpenAI-compatible model)
- **Knowledge bases** — upload documents and query them via RAG (retrieval-augmented generation) backed by pgvector
- **Model catalog** — configure which models are available and set per-model defaults
- **Multi-user** — invite-only (`selfhosted` mode) or open signup (`saas` mode)

## Architecture

```
┌─────────────────────┐     ┌──────────────────────────┐
│  Browser (Frontend) │────▶│  FastAPI Backend (:8000)  │
│  React + TanStack   │     │  LangChain (in-process)   │
└─────────────────────┘     └──────────┬───────────────┘
                                        │
                           ┌────────────▼──────────────┐
                           │  PostgreSQL + pgvector     │
                           │  (chat history + vectors)  │
                           └───────────────────────────┘
                           ┌───────────────────────────┐
                           │  Redis (optional)          │
                           │  future background queues  │
                           └───────────────────────────┘
```

The backend calls LLM providers directly in-process via LangChain — no LiteLLM sidecar, no separate worker process. Document ingest runs on the API request path (thread-offloaded), not via a Redis queue.

# Prerequisites

Before installing Vortex, ensure your server meets the following requirements.

## System requirements

| Requirement | Minimum | Recommended |
|---|---|---|
| OS | Ubuntu 22.04 / Debian 12 / any Linux with Docker | Ubuntu 22.04+ |
| RAM | 4 GB | 8 GB |
| CPU | 2 cores | 4 cores |
| Disk | 10 GB | 20 GB+ (more for large knowledge base uploads) |
| Docker | 24.0+ with Compose v2 | latest |
| Python | 3.12+ (manual install only) | 3.12 |
| Node.js | 20+ (manual install only) | 20 LTS |

## External services

### Database (required)

PostgreSQL 15+ with the **pgvector extension** enabled. This is required for knowledge base functionality (RAG).

import { Callout } from 'nextra/components'

<Callout type="info">
The Docker Compose path uses `pgvector/pgvector:pg17` which includes the extension pre-installed. For manual installs, see [pgvector installation](https://github.com/pgvector/pgvector#installation).
</Callout>

### LLM API key (required — at least one)

Vortex does not bundle a model. You must supply at least one API key:

| Provider | Environment variable | Get a key |
|---|---|---|
| Anthropic (Claude) | `ANTHROPIC_API_KEY` | [console.anthropic.com](https://console.anthropic.com) |
| OpenAI / compatible | `OPENAI_API_KEY` | [platform.openai.com](https://platform.openai.com) |
| Google Gemini | `GEMINI_API_KEY` | [aistudio.google.com](https://aistudio.google.com) |

### Embedding provider (required for knowledge bases)

If you plan to use knowledge bases, you need an embedding provider:

| Provider | Key | Notes |
|---|---|---|
| Voyage AI | `VOYAGE_API_KEY` | **Recommended.** `voyage-4-lite` model — 200M tokens/month free. |
| OpenAI | `OPENAI_API_KEY` | Falls back to OpenAI embeddings if `VOYAGE_API_KEY` is not set. |

<Callout type="warning">
An Anthropic API key does **not** provide embeddings. Set `VOYAGE_API_KEY` or `OPENAI_API_KEY` separately for knowledge base functionality.
</Callout>

### SMTP (required for selfhosted/saas modes)

Email is used for user verification and password reset. Any SMTP provider works (Resend, SendGrid, Mailgun, Gmail SMTP, etc.).

Not required in `dev` mode.

## Optional

- **Redis 7+** — not required today; Vortex runs ingest in-process. Included in Docker Compose for future use.
- **Firecrawl / Jina API key** — for the `fetch_webpage` tool (web scraping). The tool works without these via a fallback chain.

## Next step

[Quickstart →](/getting-started/quickstart)

# Quickstart

The fastest path to a running Vortex instance using Docker Compose.

import { Callout, Steps } from 'nextra/components'

<Steps>

### Clone the repository

```bash
git clone https://github.com/your-org/vortex.git
cd vortex
```

### Create `config.yaml`

Place `config.yaml` at the **repo root** (same directory as `docker-compose.yml`):

```yaml
server:
  deployment_mode: selfhosted   # or: dev (no auth), saas (open signup)
  cors_origins: http://localhost:3000

database:
  url: postgresql+psycopg://postgres:postgres@127.0.0.1:5434/ai_portal

auth:
  secret_key: "replace-with-a-long-random-secret"   # generate: python -c "import secrets; print(secrets.token_hex(32))"

smtp:
  host: smtp.example.com
  user: you@example.com
  password: "your-smtp-password"
  email_from: noreply@example.com

llm:
  anthropic_api_key: "sk-ant-..."   # or openai_api_key / gemini_api_key

embedding:
  voyage_api_key: "pa-..."   # recommended; omit to use OpenAI embeddings
```

<Callout type="warning">
`auth.secret_key` is required when `deployment_mode` is `selfhosted` or `saas`. Generate one with:
```bash
python -c "import secrets; print(secrets.token_hex(32))"
```
</Callout>

### Start the database and Redis

```bash
docker compose up -d
```

### Start the API and frontend

```bash
docker compose --profile full up -d api web
```

Migrations run automatically on startup (`alembic upgrade head`).

### Seed the model catalog

```bash
docker compose --profile full run --rm api seed-catalog-models
```

This populates the `catalog_models` table with known Anthropic, OpenAI, and Gemini model entries. Required for chat to work.

### Run the setup wizard

Open `http://localhost:3000/setup` and create your organization and admin account.

After completing setup, log in and start chatting.

</Steps>

| API | `http://localhost:8000` | FastAPI, OpenAPI at `/docs` |
| Postgres | `localhost:5434` | pgvector/pg17 |
| Redis | `localhost:6380` | Optional, not used by default |

# Docker Compose

Recommended installation method. Runs Vortex in Docker containers with all dependencies managed.

import { Callout, Steps, FileTree } from 'nextra/components'

## What this sets up

```
vortex/
├── docker-compose.yml         # infrastructure definition
├── config.yaml                # your configuration (gitignored in forks)
└── ...
```

The `docker-compose.yml` defines four services:

| Service | Profile | Description |
|---|---|---|
| `db` | (default) | PostgreSQL 17 with pgvector |
| `redis` | (default) | Redis 7 |
| `api` | `full` | FastAPI backend |
| `web` | `full` | React frontend |

Run `docker compose up -d` (no profile) to start only the database and Redis — useful when running the API and frontend on the host for development. Add `--profile full` to also start the API and frontend in Docker.

<Steps>

### Clone

```bash
git clone https://github.com/your-org/vortex.git
cd vortex
```

### Create `config.yaml`

Place at the **repo root** (next to `docker-compose.yml`). The API container mounts this file via `AI_PORTAL_CONFIG`.

Full reference: [Configuration Reference →](/configuration/reference)

Minimal production example:

```yaml
server:
  deployment_mode: selfhosted
  cors_origins: https://your-frontend-domain.com

database:
  url: postgresql+psycopg://postgres:postgres@db:5432/ai_portal

auth:
  secret_key: "your-secret-key-here"

smtp:
  host: smtp.resend.com
  port: 587
  user: resend
  password: "re_..."
  email_from: noreply@yourdomain.com

llm:
  anthropic_api_key: "sk-ant-..."

embedding:
  voyage_api_key: "pa-..."
```

<Callout type="info">
Note `database.url` uses `@db:5432` (the Docker service name) not `@127.0.0.1:5434` (the host-mapped port). Use the Docker service hostname inside containers.
</Callout>

### Start Postgres and Redis

```bash
docker compose up -d
docker compose ps
```

Expected output: `db` and `redis` services with status `running`.

### Build and start the API and frontend

```bash
docker compose --profile full build
docker compose --profile full up -d api web
```

Migrations run automatically via the Dockerfile CMD:
```
alembic upgrade head && uvicorn ai_portal.main:app ...
```

### Seed the model catalog

Run once after first deploy, and again when upgrading to a version that adds new models:

```bash
docker compose --profile full run --rm api seed-catalog-models
```

<Callout type="warning">
If `seed-catalog-models` fails with a validation error, run with `--skip-model-validation` to bypass the API reachability check (useful when running offline or with a custom model registry):
```bash
docker compose --profile full run --rm api seed-catalog-models --skip-model-validation
```
</Callout>

### Run the setup wizard (selfhosted mode only)

Navigate to `http://localhost:3000/setup` and create your organization and admin account.

If `DEPLOYMENT_MODE=saas`, users can register directly — no setup wizard.

</Steps>

## Verifying the install

```bash
# API health
curl http://localhost:8000/health

# Confirm migrations ran
docker compose logs api | grep "alembic"

# Check model catalog was seeded
curl http://localhost:8000/api/model-catalog | python -m json.tool | head -20
```

## Ports

| Service | Host port | Container port |
|---|---|---|
| Postgres | 5434 | 5432 |
| Redis | 6380 | 6379 |
| API | 8000 | 8000 |
| Frontend | 3000 | 3000 |

These host ports are chosen to not conflict with other common local services. The API and frontend ports can be changed via `config.yaml` (`server.port`) and the `web` service's `PORT` env var.

## Resetting the database

```bash
docker compose down -v          # removes volumes (destructive!)
docker compose up -d
docker compose --profile full run --rm api alembic upgrade head
docker compose --profile full run --rm api seed-catalog-models
```
```

Install Vortex without Docker, directly on the host. Useful when Docker is unavailable or when you want more control over the environment.

import { Callout, Steps, Tabs } from 'nextra/components'

## Requirements

- Python 3.12+

CREATE DATABASE ai_portal;
\c ai_portal
CREATE EXTENSION IF NOT EXISTS vector;
```

### Clone the repository

```bash
git clone https://github.com/your-org/vortex.git
cd vortex
```

### Create `config.yaml`

Place at `backend/config.yaml` (default location) or at a custom path set by `AI_PORTAL_CONFIG`:

```yaml
server:
  deployment_mode: selfhosted
  cors_origins: http://localhost:5173

database:
  url: postgresql+psycopg://postgres:yourpassword@127.0.0.1:5432/ai_portal

  password: "smtp-password"

  anthropic_api_key: "sk-ant-..."

embedding:
  voyage_api_key: "pa-..."
```

Alternatively, export env vars directly — they take precedence over `config.yaml`. See [Configuration Reference →](/configuration/reference).

### Install and run the backend

```bash
cd backend
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -e .

# Run migrations
alembic upgrade head

# Seed model catalog
seed-catalog-models

# Start the API
uvicorn ai_portal.main:app --host 0.0.0.0 --port 8000
```

<Callout type="info">
To use a config file at a custom path: `AI_PORTAL_CONFIG=/path/to/config.yaml uvicorn ...`
</Callout>

### Install and run the frontend

```bash
cd frontend
pnpm install
pnpm build
pnpm start          # production server on :3000
```

For development with hot reload:
```bash
pnpm dev --host     # starts on :5173, exposes on local network
```

### Run the setup wizard (selfhosted mode only)

Navigate to `http://localhost:3000/setup` and create your org and admin account.

</Steps>

## Running as a system service

Example `systemd` unit for the API (`/etc/systemd/system/vortex-api.service`):

```ini
[Unit]
Description=Vortex API
After=network.target postgresql.service

[Service]
Type=simple
User=vortex
WorkingDirectory=/opt/vortex/backend
Environment=AI_PORTAL_CONFIG=/opt/vortex/config.yaml
ExecStart=/opt/vortex/backend/.venv/bin/uvicorn ai_portal.main:app --host 0.0.0.0 --port 8000
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

Enable and start:
```bash
systemctl enable vortex-api
systemctl start vortex-api
systemctl status vortex-api
```
```

# How Configuration Works

Vortex can be configured in two equivalent ways. **Environment variables take precedence** over `config.yaml`.

import { Callout } from 'nextra/components'

## Option 1: `config.yaml` (recommended for self-hosters)

A structured YAML file with named sections. Default location: `backend/config.yaml`.

Override the path with the `AI_PORTAL_CONFIG` environment variable:
```bash
AI_PORTAL_CONFIG=/etc/vortex/config.yaml uvicorn ai_portal.main:app ...
```

## Option 2: Environment variables

Every setting has a flat `UPPER_CASE` env var alias. Useful for Docker, systemd, or `.env` files.

## Priority order

```
Environment variables  →  config.yaml  →  built-in defaults
       (highest)                              (lowest)
```

# Embeddings

Embeddings are required for knowledge base ingestion and RAG (retrieval-augmented generation). Without an embedding provider, document uploads will fail.

import { Callout } from 'nextra/components'

## How embeddings work

When a document is uploaded to a knowledge base:
1. The text is split into chunks
2. Each chunk is sent to the embedding provider to generate a vector
3. Vectors are stored in PostgreSQL via pgvector (`vector(1024)`)

At query time, the user's question is embedded and compared against stored vectors using cosine similarity.

## Option A — Voyage AI (recommended)

Set `VOYAGE_API_KEY`. No other configuration needed.

```yaml
embedding:
  voyage_api_key: "pa-..."
```

Vortex uses `voyage-4-lite` by default:
- 1024-dimensional vectors (matches the pgvector column)
- **200M tokens/month free** on the Voyage free tier
- Lowest-cost Voyage text embedding model

To use a different Voyage model:
```yaml
embedding:
  voyage_api_key: "pa-..."
  model: voyage-4
```

## Option B — OpenAI-compatible

If `VOYAGE_API_KEY` is not set, Vortex falls back to OpenAI-compatible embeddings using `OPENAI_API_KEY`:

```yaml
llm:
  openai_api_key: "sk-..."
  openai_api_base: https://api.openai.com/v1    # or compatible endpoint

embedding:
  model: text-embedding-3-small    # optional; this is the default
```

Vortex requests 1024 dimensions from OpenAI models to match the pgvector column.

## Choosing a provider

| | Voyage AI | OpenAI |
|---|---|---|
| Free tier | 200M tokens/month | Limited (pay-as-you-go) |
| Setup | One key (`VOYAGE_API_KEY`) | Uses your `OPENAI_API_KEY` |
| Default model | `voyage-4-lite` | `text-embedding-3-small` |
| Recommendation | ✅ Preferred | ✅ Good if already using OpenAI |

<Callout type="warning">
**Do not switch embedding providers after ingesting documents.**

Vectors from different models are not comparable. If you switch from Voyage to OpenAI (or vice versa), you must delete and re-ingest all knowledge base documents. The vectors stored for old documents will return incorrect similarity scores.
</Callout>
```

# Auth & Deployment Modes

Vortex has three deployment modes controlled by `DEPLOYMENT_MODE` (or `server.deployment_mode` in config.yaml).

import { Callout } from 'nextra/components'

## Deployment modes

### `dev` — local development only

```yaml
server:
  deployment_mode: dev

auth:
  dev_bearer_token: devtoken
  dev_seed_user_email: dev@localhost
```

- All API routes accept a fixed bearer token (`DEV_BEARER_TOKEN`)
- A seed user is auto-created on startup
- No `SECRET_KEY` required
- CORS is fully open (`allow_origins: ["*"]`)
- **Never use in production**

### `selfhosted` — single organization, invite-only

```yaml
server:
  deployment_mode: selfhosted

auth:
  secret_key: "your-secret-key-here"

smtp:
  host: smtp.example.com
  # ...
```

- Requires `SECRET_KEY` for JWT signing
- On first boot with an empty database: **all API routes return `503 Setup Required`** except `/health`, `/setup`, and `/auth/login`
- Admin completes setup at `/setup` to create the org and first user
- After setup: invite-only registration
- SMTP required for email verification and password reset

#### First-boot flow

```
Deploy with DEPLOYMENT_MODE=selfhosted
        ↓
All API routes → 503 (except /health, /setup, /auth/login)
        ↓
Navigate to https://your-app/setup
        ↓
Create org name + admin email + admin password
        ↓
Instance live — log in with admin credentials
```

### `saas` — open signup, multi-tenant

```yaml
server:
  deployment_mode: saas

auth:
  secret_key: "your-secret-key-here"

smtp:
  host: smtp.example.com
  # ...
```

- Requires `SECRET_KEY`
- Open registration via `/auth/register` and `/auth/login`
- SMTP required for email verification
- No setup wizard — users self-register

## Microsoft Entra (legacy)

`AUTH_MODE=entra` enables JWT validation via Microsoft Entra (Azure AD). This is the legacy auth mode for enterprise deployments.

```yaml
auth:
  mode: entra
  entra_tenant_id: "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
  entra_api_audience: "api://your-api-client-id"
  portal_api_key_pepper: "long-random-secret"
```

<Callout type="warning">
`auth.portal_api_key_pepper` is **required** when `auth.mode=entra`. It is used as an HMAC key to hash portal API keys (`aip_…`). Set a long random secret.
</Callout>

Frontend env vars for Entra:

VITE_ENTRA_SPA_CLIENT_ID=<SPA app registration client ID>
VITE_ENTRA_TENANT_ID=<tenant UUID>
VITE_ENTRA_API_SCOPE=<full scope string, e.g. api://<id>/access_as_user>

The `web_search` tool in chat uses a configurable search provider. Configure it with `search.provider`.

import { Callout } from 'nextra/components'

## Providers

| Provider | `search.provider` value | Key required | Notes |
|---|---|---|---|
| DuckDuckGo | `duckduckgo` | No | Default. Free, no key needed. Rate-limited for heavy use. |
| Tavily | `tavily` | `TAVILY_API_KEY` | Best quality for AI-augmented search. |
| Serper | `serper` | `SERPER_API_KEY` | Google Search via Serper API. |
| Exa | `exa` | `EXA_API_KEY` | Neural / semantic search. |

```yaml
search:
  provider: tavily
  tavily_api_key: "tvly-..."
```

## Fallback behavior

<Callout type="info">
If `search.provider` is set to a provider that requires an API key but the key is not set, Vortex **automatically falls back to DuckDuckGo** and logs a warning at startup. No error is thrown.
</Callout>

This means misconfigured search degrades gracefully rather than breaking chat entirely.

## Choosing a provider

- **DuckDuckGo** — zero setup, good for low-volume or development use
- **Tavily** — recommended for production; purpose-built for AI applications, cleaner results
- **Serper** — use if you need Google Search quality and have an existing Serper account
- **Exa** — use for research-oriented queries that benefit from semantic matching

# Fetch Providers

The `fetch_webpage` tool fetches web page content for use in chat. Vortex uses a **provider chain** — it tries each provider in order and returns the first successful result.

## Provider chain

| Priority | Provider | Key required | When active |
|---|---|---|---|
| 1 | **Firecrawl** | `FIRECRAWL_API_KEY` | Only when key is set |
| 2 | **Crawl4AI** | None | When `crawl4ai` Python package is installed |
| 3 | **Jina Reader** | Optional (`JINA_API_KEY`) | Always |
| 4 | **requests** | None | Always (last resort) |

## Providers explained

**Firecrawl** — Hosted service. Best at bypassing Cloudflare and bot protection. Returns clean markdown. Requires an API key from [firecrawl.dev](https://firecrawl.dev).

```yaml
fetch:
  firecrawl_api_key: "fc-..."
```

**Crawl4AI** — Local Playwright-based scraper with stealth mode. Handles JavaScript-heavy pages without an API key. Active only if the `crawl4ai` package is installed in the Python environment:

```bash
pip install crawl4ai
playwright install chromium
```

**Jina Reader** — Proxies requests through `r.jina.ai` to return clean markdown. Always active; no key required. Set `JINA_API_KEY` to increase rate limits:

```yaml
fetch:
  jina_api_key: "jina_..."
```

**requests** — Plain HTTP GET. Always present as a last resort. Fails on JavaScript-heavy pages and Cloudflare-protected sites.

## Recommended setup

For production with reliable scraping:

```yaml
fetch:
  firecrawl_api_key: "fc-..."    # best bypass, handles Cloudflare
  jina_api_key: "jina_..."       # higher rate limits as fallback
```

No configuration is required beyond setting keys — Vortex builds the chain automatically.

# Find the uploads directory

# Troubleshooting

Common issues and how to fix them.

import { Callout } from 'nextra/components'

## `503 Setup Required` on all routes

**Cause:** `DEPLOYMENT_MODE=selfhosted` and no organization has been created yet.

**Fix:** Navigate to `https://your-app-url/setup` to run the setup wizard and create the first org and admin account.

The following routes are always accessible even in the uninitialized state: `/health`, `/setup`, `/auth/login`.

---

## Chat returns no response / empty stream

**Cause:** No LLM API key is configured, or `CHAT_DEFAULT_API_MODEL` points to a model whose provider key is missing.

**Fix:**
1. Check that at least one LLM key is set: `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, or `GEMINI_API_KEY`
2. Check that `CHAT_DEFAULT_API_MODEL` matches a provider you have a key for. E.g. `claude-sonnet-4-6` requires `ANTHROPIC_API_KEY`.
3. Check API logs: `docker compose logs api | tail -50`

---

## Knowledge base upload fails

**Cause:** No embedding provider is configured.

**Fix:** Set either `VOYAGE_API_KEY` (recommended) or `OPENAI_API_KEY`. An Anthropic API key alone does not provide embeddings.

Look for this in the logs:
```
Set VOYAGE_API_KEY for Voyage embeddings (recommended), or OPENAI_API_KEY for OpenAI-compatible embeddings
```

---

## CORS errors in the browser

**Cause:** `CORS_ORIGINS` does not include the frontend URL.

**Fix:** Set `CORS_ORIGINS` to include all frontend origins, comma-separated:
```yaml
server:
  cors_origins: https://app.yourdomain.com,https://www.yourdomain.com
```

---

## Migration errors on startup

**Symptoms:** API fails to start with an Alembic or psycopg error.

**Possible causes:**

1. **Database not reachable** — check `DATABASE_URL` and that Postgres is running:
   ```bash
   docker compose ps
   curl -s "postgresql://postgres:postgres@localhost:5434/ai_portal" || echo "unreachable"
   ```

2. **pgvector extension not installed** — the first migration (`001_vector_extension`) creates the extension. If it fails, ensure your Postgres instance has `pgvector` available. Use the `pgvector/pgvector:pg17` Docker image.

3. **Migration revision conflict** — if upgrading from an old clone, check for column name mismatches:
   ```bash
   docker compose exec db psql -U postgres ai_portal -c "\d catalog_models"
   ```
   The `api_model_id` column must exist (renamed from a legacy name in older versions). Run `alembic upgrade head` from inside the backend.

---

## `SECRET_KEY must be set` error

**Cause:** `DEPLOYMENT_MODE` is `selfhosted` or `saas` but `SECRET_KEY` is empty.

**Fix:** Generate and set a secret key:
```bash
python -c "import secrets; print(secrets.token_hex(32))"
```
Add the output as `auth.secret_key` in `config.yaml` or as the `SECRET_KEY` env var.

---

## Search falls back to DuckDuckGo unexpectedly

**Cause:** `SEARCH_PROVIDER` is set to `tavily`/`serper`/`exa` but the corresponding API key is missing.

**Fix:** Set the API key, or explicitly set `SEARCH_PROVIDER=duckduckgo` if you don't need a paid provider.

Check logs for:
```
search_factory: tavily selected but TAVILY_API_KEY not set — falling back to DuckDuckGo
```

---

## Checking logs

```bash
# Docker Compose
docker compose logs api --tail=100 -f
docker compose logs web --tail=50

# Manual / systemd
journalctl -u vortex-api -n 100 -f
```
```
